Router Configuration

Loopbacks Â· Sub-interfaces Â· WAN Links

Router-on-a-stick: one physical port handles all VLANs via sub-interfaces. The parent e0/0 has no IP â€” it just carries frames. Each sub-interface strips the VLAN tag and routes that VLAN's traffic. This is why the switch must use a trunk toward the router.

Loopback interfaces are virtual â€” never connected to a cable, never go down. We assign one IP per router and use it as the OSPF router-id so OSPF has a stable identity even if a physical link fails.

The parent e0/0 must have no IP address AND be brought up with no shutdown. If the parent is administratively down, every sub-interface is also down â€” pings to the gateway fail with no obvious reason.

OSPF Configuration

Dynamic Routing Â· Area 0 Â· All Networks Advertised

OSPF lets routers automatically share routing information. Without it, R1 has no idea 142.100.65.0/24 (Site-B) even exists. With OSPF, every router learns every remote network automatically.

The network command tells OSPF which interfaces to advertise. Miss one and remote sites can't reach that network. Include LAN, WAN, and loopback.

passive-interface stops OSPF sending hello packets toward PCs. There is no OSPF router on the PC side so sending hellos there wastes bandwidth.

The wildcard mask is the inverse of the subnet mask. /24 â†’ wildcard 0.0.0.255. /30 â†’ wildcard 0.0.0.3. Using the subnet mask instead of the wildcard is one of the most common OSPF mistakes.

5 Real Problems I Hit Â· Exact Fixes

These are not hypothetical problems â€” these are the exact things that broke during my own lab session, in the order they happened. This is the section I most wanted before I started.

PROBLEM 1 Pings failing â€” PC could not reach its own gateway

After configuring everything, PC1 could not ping 142.100.64.1 â€” same subnet, all dots, 0% success. The cause was using VPCS syntax on an IOL router. The command ip 142.100.64.11/24 142.100.64.1 returned "Invalid input detected" and nothing was actually configured.

PROBLEM 2 SW2 trunk only allowing VLAN 100 instead of VLAN 200

PC2 could not reach its gateway even after fixing the PC IP issue. Running show interfaces trunk on SW2 showed "Vlans allowed on trunk: 100" â€” a leftover from a previous config attempt. The wrong VLAN was typed: switchport trunk allowed vlan 100 instead of 200.

Run show interfaces trunk on every switch. The allowed VLAN must match the site VLAN. Fix: switchport trunk allowed vlan 200.

PROBLEM 3 Sub-interface down even though config looked correct

E0/0.100 showed down/down in show ip interface brief despite correct encapsulation and IP. Root cause: the parent e0/0 was administratively down â€” no shutdown was not run on it. A sub-interface cannot come up if its parent is down.

Always run no shutdown on parent e0/0. Verify with show ip interface brief â€” the parent must show up up before sub-interfaces can come up.

PROBLEM 4 VLAN names wrong â€” "vlan1" instead of "HQ-VLAN"

In my first attempt the VLAN names were vlan1, vlan2, vlan3 instead of the required HQ-VLAN, SITE-B-VLAN, SITE-C-VLAN. The assignment rubric checks exact names.

Rename each VLAN: vlan 100 â†’ name HQ-VLAN. Verify with show vlan brief â€” the name column must match exactly.

PROBLEM 5 Loopbacks unreachable from PCs even after inter-site pings worked

OSPF was FULL on all routers and inter-site pings were working, but pinging loopback addresses from PCs failed. One router was missing the loopback network statement in OSPF.

Every loopback must have its own network statement. For R1: network 142.1.64.0 0.0.0.255 area 0. Check all three routers have this.
